Meeting Minutes
===============
We met on Wednesday 10th April 2024, and had the following discussions...

Current Progress on FhY:
------------------------
Today we reviewed our current progress on developement of FhY, as follows:

    1. Full Initial Implementation of the AST Nodes.
        A. Core AST Abstractions:
            a. Can be Found in the base module of the ast module
    2. Conversion of Concrete Syntax Parse Tree to AST through a Listener Class, leveraging the ASTBuilder Class.
        A. Current Status: In Progress, Not all AST Nodes are supported in Listener + Builder.
    3. Simple Unit Testing Paradigms for Test Driven Development.


Design Decisions Made or Discussed During the Meeting:
------------------------------------------------------

    1. Regarding ASTNodes: We Keep our Classes Light (i.e. Not Inheriting from third party Graph Library). All Nodes are Derivitives of the class `ASTNode`, which is just a python Abstract Base Class. We observe the same, design paradigm from `TVM <https://github.com/apache/tvm/blob/main/python/tvm/runtime/object.py#L49C1-L49C26>`_
    2. Visualization: We need more visualization tools, so we need to consider supporting GraphViz (dot language).
        A. We are currently considering a graph library which can support visualization, and graphviz.
    3. Serialization: We need to construct Methods to serialize the AST Representation, and all subsequent data structures,
        for easier debugging purposes. We should support a json.dump like interface, which can accept an indent, such that
        when we traverse subsequent child nodes, the depth of current position correctly indents the serialized data representation output.
    4. `Type Qualifiers <#Where-to-Define-Supported-Type-Qualifiers>`: We currently define supported Type Qualifiers using an enumeration in our code.
        A. Small Note: We currently only support a single Type Qualifier. but we will need to Support a list of Type Qualifiers.
    5. Type Registration: We lightly touched on the subject of registering new data types and type qualifiers. We will need a reliable way for clients to register new type definitions, much like the `Bring Your Own DataType <https://tvm.apache.org/2020/09/26/bring-your-own-datatypes#:~:text=The%20Bring%20Your%20Own%20Datatypes%20framework%20allows%20users%20to%20claim,alongside%20TVM's%20normal%20datatype%20facilities.>`_ Model.


Where to Define Supported Type Qualifiers
=========================================
Currently, the FhY Grammer does not define Type Qualifier Keywords, but instead will grab any Identifier.
Instead, we define supported Type Qualifiers (keywords) using an enumeration in our code. The rationale behind this choice
is this: Our code Raises Syntax Errors, not the parser generator, Antlr. For this reason, we will define our supported Type Qualifiers in our code. Other approaches to this problem, may create ambiguities to the FhY Language.
